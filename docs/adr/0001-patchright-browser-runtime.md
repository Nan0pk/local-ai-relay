# ADR 0001: Patchright as the Chromium automation baseline

**Evidence.** The repository contains one successful authenticated ChatGPT E2E run in a visible Fedora systemd session and nine pending provider reports. Claude has one headless empty-profile smoke result: one expected `login_required`, with zero `captcha`, `layout_changed`, `rate_limit`, or `quota_exhausted` failures. Gemini, DeepSeek, Z.ai, MiniMax, Kimi, Qwen, Grok, and Mistral record no live failures because authenticated E2E has not run. No local `~/.local-ai-relay/diagnostics/` directory was available in the review workspace. Gemini alone uses Google-backed authentication; the other providers use standalone or provider-specific authentication. The intended Linux service is headful and receives the desktop display environment, so attaching to an already-open browser is possible only in a narrower interactive workflow, not as the service default.

**Decision.** Use Patchright 1.61.1 as the shared Chromium-compatible runtime for all ten drivers, retaining the existing isolated persistent profiles, manual login, `BrowserFailure` taxonomy, and headful default. Auto-detected Google Chrome launches through `channel: "chrome"`; explicit executables and Chromium remain supported. This is baseline hardening against automation-artifact false positives, not evidence that any pending provider is fixed or E2E verified. `rate_limit` and `quota_exhausted` remain legitimate provider outcomes and receive no retry, stealth, proxy, or bypass behavior. Gemini must still be validated separately because Google session/device-trust failures are not equivalent to CDP fingerprinting.

**Rejected alternatives.** CDP attachment to an everyday signed-in browser was rejected as the default because it depends on a live desktop browser, removes profile isolation, and does not fit reliable systemd startup. Camoufox and nodriver were rejected because no provider has a recorded detection failure after Patchright, so a Firefox compatibility pass or second automation stack would be premature. Managed browser infrastructure was rejected because it contradicts the local-first trust boundary and adds cost and external session custody. CAPTCHA solvers, proxy rotation, quota evasion, and automated credential extraction remain explicitly out of scope.
